## 一、目标与定位

- 日终专家:每日为每个孩子生成一份面向家长的日报,体现"持续 / 记忆"感。
- **离线批处理**(非用户等待路径),延迟与多轮成本不敏感 —— 这是后面 agentic 方案成立的前提。

## 二、触发与时间口径 ✅

- **触发时间 04:05**(逻辑日切换后留 5 分钟余量)。
- **采用 4 点逻辑日**(`boundary_hour=4`):逻辑日 D 覆盖 [D 04:00, D+1 04:00);复用 `time.py` / session 域已有的 4 点切日 wrapper,不另造。
- `report_date` = **刚结束的那个逻辑日**。
- 深夜对话归入当晚那一天,贴合家长心智。

## 三、Worker 重构 ✅

- 把 ARQ 入口从 `app/domain/audit/worker.py` 抽到**中性的 `app/worker.py`**(与 `main.py` 同级)。
- `WORKER_SETTINGS.functions = [run_audit, run_daily_reports]` + `cron_jobs`(04:05 触发 `run_daily_reports`)。
- 域归属(守 D-1):`run_audit` 留 audit 域,`run_daily_reports` 放 expert 域;`app/worker.py` 仅做聚合装配(类比 `core/models.py`)。
- docker-compose service `audit_worker` → `worker`;command / healthcheck → `app.worker.WORKER_SETTINGS`。
- **涟漪**:M9.5 e2e burst worker 引用、其余 `app.domain.audit.worker` 的测试引用都要同步改。

## 四、overall_status 判定 ✅

- 默认 **LLM 判断**。
- **唯一硬地板**:任一轮 `crisis_detected=True` → 强制 `alert`。

## 五、架构方向:走 audit 同款 agentic(取代 json output)✅

- **输出即工具**:`ExpertReportSchema` 作为 bound tool(对应 audit 的 `AuditOutputSchema`),LLM 末步调它给结论。**取代 json output。**
- **图结构**(可结构化克隆 `audit/graph.py`):`load_context → expert_llm_call ↔ expert_tools → write_results`。
    - **拓扑保持扁平是刻意的**:agentic loop 要求 LLM 运行时自由选工具,无法把"各数据源"静态拆成并列图节点;`expert_tools` 必须是**单个薄派发节点**(按 tool name 路由 + 异常 → error ToolMessage),回边到 LLM。多加图节点反而破坏 loop、过度设计。
    - **膨胀风险在代码层、不在拓扑**:派发节点保持薄,真正逻辑下沉三层 —— 见 §8.0。
- **终止 / 降级**沿用 audit:`tool_choice="auto"` + system prompt 强约束 + post-processing 追问 + `max_iter` 降级。
- **pydantic 是契约闸**:校验 output 工具 args,失败发带字段级 `validation_errors` 的 error ToolMessage → 下一轮修正;"3 次收敛"落在 loop + `max_iter` 上,不另写重试计数。
- **降级产物**:`overall_status` 用代码地板(crisis→alert)、`content`=fallback、`degraded=True`。
- **复用**:`core/llm.py` 的 `build_role_primary` / `build_role_fallback` + `wrap_resilience`;expert 检索工具是**只读 DB 查询**,比 audit 要改 state 的 `ReplaceInNotes` 更简单。
- **不跑探测**(audit 已证明该路径),直接接 + pydantic 兜底。

**💡 建议(落地确认)**

- 新增 `Role.EXPERT`(独立 prompt / 工具绑定),topology `ROLES` 加一条。
- 思考模式**开**、`reasoning_effort=max`。
- `max_iter = 3`(expert 工具比 audit 少)。

## 六、LLM 输入材料 ✅

- **数字不喂 LLM**:6 维 `dimension_scores` 只走代码 / UI(`dimension_summary` 聚合、雷达图、跨日对比)。
- **预填(今日)**:带时间戳的 `turn_summary` 时间线(join `messages.created_at` 补时间)+ 各 session 的 `session_notes` + crisis 标记。
- **不做代码侧时段聚类**:模板化的机器描述会让家长审美疲劳;把时间线整条交给 LLM 自行组织"情绪变化 / 异常时段"。

## 七、content 结构 ✅

- **6 段结构化字段**(pydantic schema,非单块 markdown;前端 + 校验更友好):
    1. 今日概览
    2. 聊了什么
    3. 情绪变化
    4. 值得关注
    5. 具体建议
    6. 异常时段标注

## 八、工具 harness(核心)

### 8.0 代码分层(防单节点膨胀)✅

工具虽隐含多数据源,但**图拓扑不变**,膨胀靠代码分三层化解:

1. **派发节点 `expert_tools` (薄)**:仅按 tool name 路由 + 统一异常 → error ToolMessage,不写查询逻辑。
2. **工具处理层 `expert/tools.py`**:`search_history` / `fetch_by_ref` 两个 handler,只做编排 —— `search_history` 按 `sources` 扇出、合并、排序、截断;`fetch_by_ref` 解析 ref、按 kind 派发。
3. **数据源仓储层 `expert/repository.py`**:每个数据源一个只读查询函数(`search_turn_summaries` / `search_session_notes` / `search_crisis_topics` / `search_daily_reports`;`fetch_turn` / `fetch_notes` / `fetch_report`(`fetch_turn` 内部用 `load_turn_messages` 取该轮原文))。新增 / 调整数据源只动这层。

### 8.1 安全地基 ✅

- `child_user_id` 走 `ExpertContextSchema` 注入,**绝不进工具参数**(防 LLM 编造他人 id 串数据)。
- `owned_session_ids`(该孩子的 session id 集合)**与 `child_user_id` 同属运行不变量,一并进 `ExpertContextSchema`(DI)**:由 usecase 在建图前一次性查好注入,工具从 Runtime 读取做**内存校验**,不再每次 `JOIN sessions`;`report:` 引用直接比 `child_user_id`。
- **state 只留可变工作数据**(message 列表 / `tool_iter_count` / `structured_output` 等),不变量一律进 context —— 同 audit 纪律(`child_profile` 也是建图前载入 context)。
- 前提:集合在单次 run 内稳定 —— 成立,日报只覆盖刚结束的逻辑日,04:05 后新建 session 属次日、不在检索窗口。

### 8.2 ① 历史日终报告 → 预填,不做工具 ✅

近 N 天日终报告(date + status + 概览)直接进 `load_context`,不占工具轮次。

### 8.3 ② `search_history`(检索 → 线索 + 引用)✅

- 入参:
    - `keywords`(必填,1–8 个,每个 ≥2 字):**多词 OR 匹配**(`ILIKE ANY`),鼓励 LLM 一次给近义 / 相关词做 query expansion,补 ILIKE 无语义召回的短板。
    - `start_date` / `end_date`(可选,**逻辑日** ISO date;默认 `end_date`=report_date 前一日、`start_date`=往前 30 日):代码转 UTC 窗口比对 `created_at`。校验 start≤end、end<report_date(不含今天)、跨度 clamp ≤90 日。
    - `limit=15`(clamp [1,50])、`context_chars=100`(clamp [0,300])、可选 `sources`。让 LLM 凭精确时间窗 + 条数自行控召回与过滤。
- **可检索源(出处分类,4 类)**:`turn_summary` / `crisis_topic` / `session_notes` / `daily_report`(仅搜预填窗口之外)。**原始 message 不进检索**(噪声大、灌爆上下文)。
- 每条命中返回:

| 字段 | 含义 |
| --- | --- |
| `ref` | 引用 id(给 ③ 用) |
| `source` | 出处分类(4 类之一) |
| `snippet` | 命中段:短源(turn_summary / crisis_topic)整段返;长源(session_notes / daily_report)按 `context_chars` 前后开窗 |
| `occurred_at` | turn_summary → 该轮 ai message `created_at`;crisis_topic → `audit_records.created_at`;session_notes → 该 session 首末消息**区间**(滚动文档无单点时间);daily_report → `report_date` |
| `matched` | 命中了 `keywords` 里的哪些词(供 LLM 自判相关性与排序) |
| `locating` | `session_id` / `turn` / `report_date` 等人读定位信息 |
- 机制:多词 `ILIKE ANY` 扫上述文本层(`turn_summaries` 用 `jsonb_array_elements` 后比 `->>'summary'`),命中任一词即返,`occurred_at` DESC 排序后 `limit` 截断。
    - **为何不走正则**:中文无空格、词边界 `\b` 基本失效,正则收益有限;LLM 写的正则易错、有 ReDoS(灾难性回溯)拖垮 DB 的风险,对 JSONB 逐元素跑成本也高 —— **多词 OR 已覆盖"猜不准关键字"的主要诉求,更安全可控**。
    - **局限**:仍是字面子串(多词 OR 靠 LLM 列同义词补),无真正语义召回;要更准 → 上 embeddings,而非正则。

### 8.4 ③ `fetch_by_ref`(按引用 → 精确)✅

- **引用 kind**:`turn` / `notes` / `report`。原 `audit:` 独立 kind **已并入 `turn`**(分数 / 危机在 `audit_records`、摘要在 `rolling_summaries`,但轮级信息统一由 `turn:` 一把取全)。**拿原文不单独寻址 message —— 原文随 `turn:` 返回**(message_id 不对 LLM 暴露)。
- **引用格式(半结构化、可自建 — 💡 建议采纳)**:LLM 既能回填 ② 的结果,也能用预填里的 `session_id` + `turn` **自己拼引用**直接 drill 今日,无需先白跑检索;安全靠 8.1 内存校验兜住。严格解析,非法 → error ToolMessage。
    - `turn:{session_id}#{turn}` —— **拿原文就用它**(返回的 bundle 含该轮 human / ai 原文,见下表)
    - `notes:{session_id}`(整段返,**无 section**)
    - `report:{report_id}`
- 入参:`ref`、`context_turns=0`(clamp [0,3],仅 `turn` 类展开前后邻轮)。
- **返回原则(💡 建议)**:单元内返全;跨单元不默认扩,要看邻轮才显式传 `context_turns` —— 防上下文悄悄膨胀。

| 引用 kind | 返回 |
| --- | --- |
| `turn:` | turn_summary 全文 + 该轮 human / ai **原文**  • crisis 标记 / topic;**不含 `dimension_scores`**(数字不进 LLM);`context_turns>0` 附前后各 N 轮原文 |
| `notes:` | 该 session `session_notes` 全文 + 元信息(date 区间 / 轮数) |
| `report:` | 该日终报告完整 content |

### 8.5 提示词"记忆"工作流 ✅

- 写"具体建议"前 → 用 ① 预填的历史报告看趋势("比上周缓和""第三次提到")。
- 今日材料出现值得追溯的话题 / 情绪 / 事件 → `search_history` 查历史,命中再 `fetch_by_ref` 回原文核实。
- **没查到就如实写"近期首次出现",禁止臆造连续性**(冷启动 / 新孩子高压线)。
